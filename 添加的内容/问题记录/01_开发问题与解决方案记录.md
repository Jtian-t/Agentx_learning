## 🚀 实战避坑 & 架构反思

### 01_Spring Security 架构错位与拦截器重构
- **问题描述**: 在 Phase 1 开始时，Agent 习惯性地准备实现 `UserDetailsService` (标准的 Spring Security 流程)，但发现原厂项目中并没有这个类。
- **定位过程**: 通过 `graphify` 全量扫描 `org.xhy` 包，发现原厂虽引入了 Security 依赖，但却使用 `WebMvcConfigurer` + `HandlerInterceptor` 实现了更轻量级的 JWT 校验。
- **解决方案**: 废弃 `UserDetailsService` 计划，转而复现 `UserAuthInterceptor`。
- **面试谈资 (Key Point)**: 
    > “在复现 AgentX 这种高并发、追求轻量化的项目时，我发现它没有直接套用 Spring Security 沉重的 Filter Chain 逻辑，而是选择了 **拦截器 (Interceptor) + ThreadLocal (UserContext)** 的方案。这让我深刻理解了**架构灵活性**的重要性——不应死守框架，而应根据业务（如 Agent 的频繁身份透传）选择最合适的鉴权深度。”

### 02_洋葱架构中的路径冗余与包规范
- **问题描述**: 在实现 `Result.java` 时，出现了 `infrastructure/infrastructure/common` 的二级冗余路径。
- **原因分析**: 自动生成包名时，重复包含了模块前缀。
- **解决方案**: 执行“手术级”重构，将 `Result.java` 移动到标准的 `infrastructure/common` 路径，并对齐原厂 DDD 结构。
- **面试谈资 (Key Point)**: 
    > “在项目初期，我对 **DDD (领域驱动设计)** 的表现层（Interfaces）与基础设施层（Infrastructure）的边界感还不够敏锐。通过对冗余路径的重构，我建立了一套严格的代码规范检查流程，确保项目具有‘原厂级’的整洁度，这对于大型分布式系统的协作至关重要。”

### 03_Spring Boot 3.x/4.x 的依赖缺失与 Servlet 升级
- **问题描述**: 在编写拦截器时，发现 `jakarta.servlet` 等核心 API 无法加载。
- **定位过程**: 检查 `pom.xml` 发现仅引入了 `spring-boot-starter` (基础容器)，未引入 `spring-boot-starter-web` (MVC 容器)。
- **解决方案**: 升级 Maven 依赖并刷新上下文。
- **面试谈资 (Key Point)**: 
    > “在 Spring Boot 3 时代，传统的 `javax.servlet` 全面升级为 `jakarta.servlet`。我在处理拦截器依赖时，快速定位了基础启动类与 Web 启动类的区别，并对 Spring Boot 的**自动配置 (Auto-Configuration)** 机制有了更扎实的理解。”
